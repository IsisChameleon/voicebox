voiceboxsynthetic-user signal lab

An MCP server that gives an LLM a voice and ears inside a browser — so Claude can play a synthetic user against any web voice app (Daily, LiveKit, raw RTCPeerConnection) without the app knowing. Local models, no API keys: Kokoro TTS in, Whisper STT out.

Audio never crosses the MCP boundary

MCP carries text / control only. The actual audio flows out-of-band over a raw-16-bit-PCM WebSocket between shim.js (inside the page) and the pipecat child. Three processes, spawned (not forked).

the bug that started it

The silent WebRTC tee

listen() returned the same Whisper silence-hallucination every call while the bot spoke for 6+s. A MediaStreamAudioSourceNode alone doesn't drive the depacketizer — the worklet ticked but forwarded all zeros.

proven empirically

Render forces decode

A loopback matrix (headless×muted) showed peak 0.0 with no sink, 1.0 with any rendering element. The e2e test only ever worked because the readme app mounts its own <audio> — an accidental side-effect.

intentional asymmetry

48 kHz out / 16 kHz in

Browser→pipecat = 16 kHz because mlx_whisper hard-assumes it (no sample_rate param); the shim resamples 48→16. pipecat→browser = 48 kHz to match the page's native AudioContext.

known limitation

Cross-origin iframe blind spot

The RTCPeerConnection wrap only reaches the top-level frame. Daily Prebuilt's <DailyIframe> lives in a daily.co-origin frame — the hook never gets there. Detect: audioTrackCount === 0 after call join. Use Daily's headless SDK instead of Prebuilt to avoid this. See tab 03 for full gap analysis.

One browser, two MCP clients, one orchestrator

voicebox owns the audio half and hands back a cdp_endpoint. Claude points a second MCP client (@playwright/mcp --cdp-endpoint) at that same Chromium to drive the UI. voicebox deliberately ships no navigate/click tools — that split is the whole design.

surprising fit

CI is actually better

A single CI runner means browser + clients all live on localhost — no client/browser context split, no cross-machine wiring. The headless synthetic-user loop runs clean.

the sharp edge

CDP context split

A connect_over_cdp client sees the page under contexts[0] but cookie ops hit the default context → come back empty. storage_state can load but can't be saved via CDP.

so instead

Session reuse via user_data_dir

launch_persistent_context uses the default context, so it's fully CDP-coherent. That's why voicebox exposes user_data_dir and dropped storage_state entirely.

Two hooks, two blind spots

shim.js is injected via page.add_init_script (page-scoped, not context-scoped) before any page code runs — it replaces global browser APIs in-place. Scoping to the voicebox tab prevents @playwright/mcp's extra tabs from connecting to the audio WS and triggering a reconnect storm. Hook 1 controls what the page sends; Hook 2 controls what we hear. Each hook has preconditions; if they fail, it skips silently.

Hook 1 · outbound — what the page sends

getUserMedia() override

When the page calls navigator.mediaDevices.getUserMedia({audio: true}) the shim intercepts and returns a synthetic MediaStream backed by a MediaStreamTrackGenerator. That generator is fed by inbound Kokoro PCM frames arriving over WS :9091. The app's RTCPeerConnection then sends our voice to the remote peer — without the app knowing.

For video+audio calls: real video tracks are preserved; only the audio track is swapped.

Preconditions (either failing → hook skipped silently)
· navigator.mediaDevices exists (unavailable on non-localhost HTTP, about:blank)
· MediaStreamTrackGenerator + AudioData exist (WebCodecs — present in Chrome ≥ 94)
· The page actually calls getUserMedia({audio}) — apps that skip mic access entirely are unaffected (Hook 2 still covers them)
Diagnostic: window.__voiceShim.micHookInstalled · inboundChunks
Hook 2 · inbound — what we hear

RTCPeerConnection wrap

The shim saves OrigRTCPeerConnection = window.RTCPeerConnection, then defines class WrappedRTCPeerConnection extends OrigRTCPeerConnection and replaces the global. The subclass approach means all native WebRTC behavior is unchanged — only a track listener is added in the constructor.

On each new inbound audio track: a muted hidden <audio> element forces Chromium to decode the track (without it, createMediaStreamSource returns silence — see the silent-tee fix). The decoded stream is routed through a Web Audio graph at 16 kHz → AudioWorkletNode → Float32-to-Int16 → raw PCM → WS :9091 → pipecat/Whisper.

Track events are deduped by track.id — Daily opens several peer connections for the same logical bot audio; without dedup, the same voice arrives 2–3× and Whisper gets an overlapping stream.

Preconditions (either failing → hook skipped silently)
· window.RTCPeerConnection exists (missing on very old browsers or non-WebRTC pages)
· AudioContext + AudioWorkletNode exist (Web Audio — available everywhere Chromium ships)
· The app constructs its RTCPeerConnection on the main thread of the top frame — see blind spots below
Diagnostic: window.__voiceShim.pcHookInstalled · pcCount · audioTrackCount · outboundChunks

Coverage gaps

Hook 2 only intercepts peer connections constructed on the main thread of the top-level frame. Three isolation boundaries can put the RTC code out of reach.

real gap · likely to hit

Cross-origin iframes

page.add_init_script runs only in the top-level browsing context of the voicebox-owned tab. A cross-origin frame — e.g. Daily Prebuilt's <DailyIframe> embedded from daily.co — has a separate JS context. The window.RTCPeerConnection replacement never reaches it; the peer connection is constructed in the frame's global, which is untouched.

Detection: after the call joins, window.__voiceShim.audioTrackCount === 0. If the page uses Daily Prebuilt, Hook 2 will be silent.

Workaround options: use the Daily SDK (headless API, not Prebuilt) which constructs PCs on the main page, or use Daily's startRecording / getNetworkTopology APIs for a server-side tap instead.

theoretical gap · dormant in practice

Web Workers

Chromium added RTCPeerConnection to the DedicatedWorker scope around v100 (~2022). If an SDK constructed its peer connection inside a Worker, the hook on window.RTCPeerConnection would not propagate there.

In practice: no mainstream voice SDK does this. Daily, LiveKit, Twilio, and Agora all construct their peer connections on the main thread. This gap is architecturally real but currently harmless for all target apps.

Detection would be the same: audioTrackCount === 0 with no cross-origin iframe present.

resolved — was a real trap

Extra-tab reconnect storm

When the shim was injected at context level (context.add_init_script), every new tab that @playwright/mcp opened also got the shim. That tab's shim connected to WS :9091 and pipecat kicked the existing connection ("only one client allowed"). The original tab reconnected after 1 s and kicked the new one. Both sides retried endlessly at ~1 Hz.

Fixed by switching to page-scoped injection (page.add_init_script). The shim lives only in the voicebox tab; new tabs opened by @playwright/mcp are plain browser tabs with no audio connection.

not a real gap

Service Workers

Service Workers are network proxies — they have no access to RTCPeerConnection, getUserMedia, or AudioContext. No voice app routes WebRTC through a Service Worker; the APIs simply don't exist there.

Mentioned for completeness: this limitation does not apply to any audio use case and can be ignored.
